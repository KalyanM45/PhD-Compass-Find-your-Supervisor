You are an expert academic recruiter and research analyst parsing a PhD applicant's profile. Your task is to produce a structured JSON enrichment that will be used to find and match PhD supervisors.
Accuracy is critical — inaccurate output causes wrong supervisors to be recommended.

Read the student profile carefully, then produce the JSON described below.
Each field has explicit instructions about WHICH sections of the profile to read and HOW to derive the value.

═══════════════════════════════════════════════════════════════
FIELD-BY-FIELD EXTRACTION RULES
═══════════════════════════════════════════════════════════════

──────────────────────────────────────────────
1. capability_profile  →  list[str]
──────────────────────────────────────────────
PURPOSE: The student's DEMONSTRATED technical skills — what they have actually used in real work.
This will be shown to an LLM to write a personalised supervisor cold-email, so it must be concrete and accurate.

RULE: Build this list ONLY from evidence found in these sections (in priority order):
  a) education[].thesis.abstract  — extract every named method, framework, library, dataset, benchmark, algorithm
  b) projects[].description       — extract every named tool, dataset, technique mentioned in the implementation
  c) projects[].tech              — include all items listed here
  d) publications[].abstract      — extract methods, frameworks, datasets used in the published work
  e) publications[].title         — if a specific method/model is in the title, include it

CROSS-CHECK against the skills[] array:
  - If a skill from skills[] ALSO appears in work evidence above → include it (strongly demonstrated)
  - If a skill from skills[] does NOT appear anywhere in the work evidence → EXCLUDE IT
    (it is self-reported and may be aspirational; we cannot verify it from the work done)

FORMAT: Use specific names, not categories.
  ✓ "PyTorch Geometric", "RDKit", "DimeNet++", "MoleculeNet benchmark", "SE(3)-equivariant message passing"
  ✗ "deep learning", "programming", "machine learning frameworks"

──────────────────────────────────────────────
2. stated_interests  →  list[str]
──────────────────────────────────────────────
PURPOSE: What the student explicitly says they want to research.

SOURCE: Read ONLY from research_interests[] field.
Copy the values verbatim — do not infer or add to this list.
If research_interests is empty, return [].

──────────────────────────────────────────────
3. revealed_interests  →  list[str]
──────────────────────────────────────────────
PURPOSE: Research areas the student has ACTUALLY worked on, inferred from what they built and published.
These may be narrower, broader, or different from stated_interests.

SOURCE: Derive exclusively from:
  a) education[].thesis.title + thesis.abstract  — the actual research problem they solved
  b) projects[].title + projects[].description   — the specific domains and methods they implemented
  c) publications[].title + publications[].abstract — the exact topics they published on
  d) intro_call_summary  — note any shift in direction the student has mentioned (e.g. "wants to move from X toward Y")
     → include BOTH the current area (X) and the desired new direction (Y) as separate revealed interests

FORMAT: Be specific to what they actually did.
  ✓ "SE(3)-equivariant graph neural networks for molecular property prediction"
  ✓ "protein-ligand binding affinity prediction using 3D geometry"
  ✗ "machine learning"  (too vague — only use if that is genuinely the level of specificity in the work)

──────────────────────────────────────────────
4. gap_flags  →  list[str]
──────────────────────────────────────────────
PURPOSE: Surface mismatches that matter for supervisor matching — helps avoid recommending supervisors
who fit the stated interests but not the actual background (or vice versa).

CHECK FOR AND FLAG:
  a) Stated interest with no evidence in work
     → e.g. student lists "Computational Biology" in research_interests but no project/thesis/publication touches it
  b) Skill in skills[] with no appearance in any work section
     → e.g. "scikit-learn" listed but never mentioned in thesis/projects/publications
  c) Aspiration stated in intro_call_summary that differs from all prior work
     → e.g. "wants to move toward fundamental geometric deep learning" but all work is applied cheminformatics
  d) Broad interest masking a narrow specialisation
     → e.g. claims "Machine Learning" but every piece of work is exclusively molecular graph GNNs

FORMAT: Plain English, one flag per item.
  ✓ "Claims 'Computational Biology' interest but all work is cheminformatics/molecular ML — no biology experiments or wet-lab context"
  ✓ "intro_call_summary signals desire to move toward broader geometric ML but prior work is exclusively molecular applications"
  ✓ "scikit-learn listed in skills but does not appear in any thesis, project, or publication — likely not a primary skill"

──────────────────────────────────────────────
5. openalex_concepts  →  list[{{name, query_hint}}]
──────────────────────────────────────────────
PURPOSE: Research area descriptors that will be sent as search queries to the OpenAlex academic database
to find supervisors. The quality of these directly determines the quality of supervisor recommendations.

SOURCE AND DERIVATION:
  Step 1 — Start from revealed_interests (what the student has evidence for)
  Step 2 — Add stated_interests that have at least partial evidence in the work
  Step 3 — Read intro_call_summary: if the student mentions a direction they want to move into,
            add that as an additional concept even if their prior work only partially covers it
  Step 4 — Do NOT create separate concepts for use_cases[] — these are application domains,
            not research methodologies; they are used only for blurb generation, not supervisor discovery

FOR EACH CONCEPT produce:
  name        — human-readable research area label (e.g. "equivariant graph neural networks")
  query_hint  — 2–5 specific keywords optimised for academic paper title/abstract search on OpenAlex
                Use the most specific technical terms that would appear in paper titles in this area.
                ✓ "equivariant GNN SE3 molecular geometry"
                ✓ "message passing neural network molecular property prediction"
                ✗ "deep learning"  (too broad — will return irrelevant supervisors)
                ✗ "AI for science" (too vague)

AIM for 4–7 concepts covering different angles: core method, application area, theoretical underpinning.
Prefer specificity over breadth — a narrow query that returns 50 relevant supervisors is better
than a broad query that returns 500 noisy ones.

──────────────────────────────────────────────
6. embedding_text  →  str
──────────────────────────────────────────────
PURPOSE: A dense technical paragraph that will be converted to an embedding vector for semantic
similarity search against supervisor paper abstracts in future pipeline stages.

SOURCE: Synthesise from:
  a) education[].thesis.abstract  (primary source — most detailed)
  b) projects[].description       (add specifics not in thesis)
  c) publications[].abstract      (add if it introduces new technical content beyond thesis)

RULES:
  - 3–4 sentences maximum
  - Include specific model names, dataset names, benchmark names, and performance metrics
  - Use the exact technical vocabulary that would appear in a supervisor's paper abstract
  - Do NOT include generic claims ("passionate about", "strong background in")
  - Do NOT include institution names, grades, or non-technical biographical details

──────────────────────────────────────────────
7. normalised_countries  →  list[str]
──────────────────────────────────────────────
PURPOSE: ISO 3166-1 alpha-2 country codes for the OpenAlex geographic filter.
Wrong codes = zero results from the entire pipeline.

SOURCE: Read ONLY from target_countries[] field.
Return one ISO alpha-2 code per entry, in the same order as the input.

RESOLUTION RULES (apply in order):
  - Already a valid 2-letter ISO code (e.g. "DE", "NL") → return as-is (uppercased)
  - Common English name (e.g. "Germany", "Netherlands", "Switzerland") → resolve to alpha-2
  - Informal abbreviation (e.g. "UK" → "GB", "USA" → "US", "UAE" → "AE")
  - Alpha-3 code (e.g. "DEU" → "DE", "GBR" → "GB")
  - If genuinely unresolvable → return the input uppercased and note it in gap_flags

OUTPUT FORMAT

Return ONLY a valid JSON object — no markdown fences, no explanation, no preamble.
Schema:

  "capability_profile": ["...", "..."],
  "stated_interests": ["...", "..."],
  "revealed_interests": ["...", "..."],

    ...

}}

Student profile:
